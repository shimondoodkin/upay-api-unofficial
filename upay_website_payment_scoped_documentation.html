Upay Website Payment Gateway — Scoped Documentation

Scope: website payment, payment lookup/verification, refunds/cancellations, and receipt/invoice creation. This version intentionally excludes direct in-app payments, broad reports, and account updates.

Integration scope

In scopeOut of scope
Website checkout form/page, returnurl/ipnurl, transaction lookup, refunds, receipts, invoicesIn-app payments, extensive reports, account/profile updates, bank/account details, contacts, settings, messages

Credential location

Upay website → login → side menu → User Details. Keep API credentials server-side only.

Website payment flow

  1. Merchant server creates local order.
  2. Merchant server generates Upay form/page on the fly.
  3. Browser posts to https://app.upay.co.il/API6/clientsecure/redirectpage.php.
  4. Customer pays on Upay hosted page.
  5. Upay redirects to returnurl and can notify separate ipnurl.
  6. Merchant server calls Upay lookup before fulfillment.
Important: hidden form fields are editable in the browser. Always verify server-side before delivering goods.

Verification

Recommended automatic paid status until Upay confirms more: S only.

on returnurl or ipnurl:
  extract order_id / transaction hint
  load local pending order
  call Upay transaction lookup
  compare amount + currency + status + reference
  reject duplicates
  save transactionid/cashierid
  mark paid once
  then create/fetch receipt/invoice

Callback/IPN trust model

Rule: returnurl and ipnurl are triggers only. The proof of payment is the authenticated server-side transaction lookup from Upay.
notification received
→ extract order_id / transactionid / cashierid hint
→ load local pending order
→ fetch Upay transaction server-to-server
→ match fetched transaction to local order
→ verify amount, currency, status, merchant/account, duplicate use
→ save Upay transactionid/cashierid
→ mark paid exactly once
Store locallyWhy
order_id, payment_referenceIdentify the local order and correlate it with Upay data.
expected_amount, expected_currencyHidden form fields can be modified in the browser.
upay_transaction_id, upay_cashieridSave after lookup and enforce uniqueness to prevent replay/double fulfillment.
upay_status, upay_verified_at, paid_atKeep an audit trail and mark paid only after verification passes.

A callback signature/HMAC is useful for reducing fake callback noise, but it is not mandatory if callbacks never directly mark orders paid.

Questions still worth verifying

  • Does Upay retry ipnurl, and on what schedule?
  • Can callback/IPN be signed with HMAC or another signature? Not required for correctness if server lookup is authoritative, but useful for noise reduction.
  • Is S final captured/settled for website card payments?
  • What exactly does A mean for website card payments?
  • Can refund API be used with API credentials, or does it require logged-in user/password MD5?
